#### FR-T2-003: Consent — Đồng ý chia sẻ thông tin

**Source:** Account Linking PRD — FR-003
**Priority:** Must Have V2

**V2 addition:** Trước khi BE thực hiện matching (mới có ở V2), FE hiển thị màn consent.

**Key Acceptance Criteria:**
- Màn consent hiển thị: Họ tên, CCCD, SĐT, Email sẽ share
- Mô tả ngắn từng item
- 2 nút: "Đồng ý chia sẻ" / "Từ chối"
- Từ chối → quay về GG, KHÔNG link
- Đồng ý → ghi `consent_at` timestamp → tiếp matching
- CCCD masked (4 số cuối)

**Migration note V1 → V2:** Flow V1 không có consent screen, sau callback OAuth là link luôn. V2 add consent screen trước khi gọi matching engine.

**Full spec:** [Account Linking FR-003](prd-account-linking-2026-05-04.md#fr-003-consent--đồng-ý-chia-sẻ-thông-tin)

**Source:** Account Linking PRD — FR-004

#### FR-T2-005: Matching CCCD

**Source:** Account Linking PRD — FR-005
**Priority:** Must Have V2

**Key Acceptance Criteria:**
- 2 bên có CCCD giống → PASS
- 2 bên có CCCD khác → REJECT (không phải cùng người)
- 1 bên có, 1 bên không → bỏ qua check, fallback SĐT/Email
- Chuẩn hóa CCCD: chỉ giữ số
- Edge cases: `"000000000000"` invalid, CCCD < 12 số invalid

**Open question:** Scalef SSO response hiện không có CCCD → cần Scalef bổ sung field hoặc skip CCCD check.

**Full spec:** [Account Linking FR-005](prd-account-linking-2026-05-04.md#fr-005-matching-cccd-identity-check)

#### FR-T2-018: Data Normalization

**Source:** Account Linking PRD — FR-018
**Priority:** Must Have V2

**Key Acceptance Criteria:**
- CCCD: bỏ space + special chars, chỉ giữ số
- SĐT: parse về `+84xxxxxxxxx`, hỗ trợ input `0xxx`, `+84xxx`, `84xxx`, có space/dash
- Email: lowercase + trim
- Tên: trim trailing spaces, KHÔNG normalize Unicode (giữ accent VN)
- Centralized utility — reuse cho matching + unique check

**Full spec:** [Account Linking FR-018](prd-account-linking-2026-05-04.md#fr-018-data-normalization)

#### FR-T2-008: OTP xác thực khi update

**Source:** Account Linking PRD — FR-008
**Priority:** Must Have V2

**Key Acceptance Criteria:**
- Trigger OTP khi user chọn SĐT/Email từ Scalef → update GG
- OTP gửi về **target mới** (Scalef value)

- Rate limit: 1 OTP / 60s / target
- Reuse `internal/service/otp.go` của vcreator

#### FR-T2-010: Bidirectional update + Saga rollback

**Source:** Account Linking PRD — FR-010
**Priority:** Must Have V2

**Key Acceptance Criteria:**
- User chọn data GG → update Scalef qua `PUT /users/{id}/profile`
- User chọn data Scalef → update GG local
- Saga: update GG local → gọi Scalef → confirm GG. Scalef fail → rollback GG
- Background retry 3 lần exponential backoff (1s, 2s, 4s)
- Audit `scalef_link_history` với changes detail

**Open question:** Scalef `PUT /users/{id}/profile` API chưa confirm.

**Full spec:** [Account Linking FR-010](prd-account-linking-2026-05-04.md#fr-010-bidirectional-update--saga-rollback)

## V2 Migration Strategy V1 → V2

**Database migrations:**
1. Add `consent_at` field to users
2. Extend `pending_scalef_link.step` enum: `oauth_callback_received`, `consent`, `compared`, `awaiting_otp`, `awaiting_email_verify`, `committed`
3. Create `scalef_link_history` collection
4. Backfill existing `scalef_user_id` users → insert `linked` history records (best effort, no `changes` data)

**Code migration:**
1. V1 OAuth callback handler → wrap thêm step "show consent screen" trước khi commit
2. V1 commit logic (`save scalef_user_id`) → wrap vào saga state machine
3. Add matching engine giữa callback và commit

**Data cleanup pre-V2 launch:**
- Scan `users` collection: tìm cases `scalef_user_id` duplicate (V1 chấp nhận) → migration plan riêng (manual review hoặc auto resolve theo timestamp)
- Identify users có data SĐT/Email lệch giữa GG-Scalef → notify cho user re-link nếu cần
